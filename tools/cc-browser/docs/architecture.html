cc-browser Architecture

Chrome Extension + Native Messaging + HTTP Daemon

Overview

cc-browser is a browser automation system built on three components: an HTTP daemon, a Chrome extension, and a native messaging host. The daemon is a single background process that manages all browser connections. Multiple clients (CC Director instances, CLI calls, other tools) talk to the same daemon over HTTP.

Key insight: One daemon serves all clients. Two CC Director windows both talk to the same daemon on port 9280. The daemon is a separate Node.js process, independent from any WPF application.

Communication Chain

When a client sends a browser command (e.g. "click element #submit"), the request flows through four layers: 

  CLIENT              DAEMON              NATIVE HOST          EXTENSION            BROWSER
  ======              ======              ===========          =========            =======

  HTTP POST -------> Receives request
  /command            on port 9280
                           |
                           | WebSocket (ws://localhost:9280/ws)
                           |
                           +----------> native-host.mjs
                                        (stdio bridge)
                                              |
                                              | Chrome Native Messaging
                                              | (stdin/stdout JSON)
                                              |
                                              +----------> background.js
                                                           (service worker)
                                                                 |
                                                                 | chrome.tabs API
                                                                 | + content scripts
                                                                 |
                                                                 +----------> Page DOM
                                                                              (click, type,
                                                                               snapshot, etc.)

  <--- HTTP response with result ---

Why This Chain?

  • HTTP daemon -- universal interface, any language/tool can call it
  • WebSocket -- persistent bidirectional channel for real-time command/response
  • Native Messaging -- Chrome's official way for local apps to talk to extensions
  • Extension -- has full access to chrome.tabs, DOM, network, cookies

Component Details

1. HTTP Daemon (daemon.mjs)

A plain Node.js HTTP server. No Express, no frameworks. Manages all state: connections, locks, action history, and the WebSocket transport layer.

  • Default port: 9280
  • Lockfile: %LOCALAPPDATA%\cc-browser\daemon.lock
  • Auto-started by CLI commands, but NOT by CC Director (known gap)
  • Single instance -- second start detects lockfile and exits

2. WebSocket Transport (transport.mjs)

Manages persistent WebSocket connections from native messaging hosts. Each browser connection registers with a name (e.g. "reddit", "linkedin"). Commands are routed to the correct browser by connection name.

  • Path: ws://127.0.0.1:9280/ws
  • One WebSocket per browser instance
  • Request/response matching via unique request IDs
  • Default command timeout: 30 seconds

3. Native Messaging Host (native-host.mjs)

A small Node.js script that bridges Chrome's native messaging protocol (length-prefixed JSON over stdin/stdout) with the daemon's WebSocket. Chrome launches it automatically when the extension connects.

4. Chrome Extension (extension/)

A Manifest V3 extension with a background service worker and content scripts. The service worker receives commands from the native host and executes them using Chrome APIs (tabs, scripting, debugger).

  • background.js -- command router, native messaging, network logging
  • content.js -- DOM interaction, ARIA snapshots, element targeting

5. CLI Client (cli.mjs)

The cc-browser command users type in terminals. Translates command-line arguments into HTTP requests to the daemon. Auto-starts the daemon if it is not running.

6. CC Director Integration (WPF)

The Connections panel in CC Director sends HTTP requests directly to the daemon. The Workflow feature opens a recorder window alongside a positioned browser.

Known gap: CC Director does NOT auto-start the daemon. If the daemon is not running, connection operations fail with "target machine actively refused it" on port 9280.

Daemon Lifecycle

  START
  =====
  1. CLI runs any command (e.g. "cc-browser connections status")
  2. CLI checks daemon.lock -- daemon not running
  3. CLI spawns: node daemon.mjs --port 9280
  4. Daemon writes daemon.lock: { port: 9280, pid: 12345 }
  5. Daemon listens on HTTP :9280 + WebSocket :9280/ws
  6. CLI connects and sends command

  RUNTIME
  =======
  - Daemon stays running indefinitely
  - Multiple clients connect via HTTP
  - Each browser connection registers via WebSocket
  - Connection locks prevent concurrent access (5min TTL)
  - Action history kept in memory (max 1000 entries)

  SHUTDOWN
  ========
  1. Daemon receives SIGINT/SIGTERM (or process killed)
  2. Closes all WebSocket connections
  3. Removes daemon.lock file
  4. Process exits

    |-- POST http://127.0.0.1:{port}/connections/open
